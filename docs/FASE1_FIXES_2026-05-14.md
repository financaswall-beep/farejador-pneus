## Os 6 fixes (em ordem de execução)

### Fix 1 — Bug D — Subir `maxTokens` do Generator + preservar tokens no catch

**Commit:** [`224f7e9`](https://github.com/financaswall-beep/farejador-pneus/commit/224f7e9)
`fix(generator): raise maxTokens to 1500 and preserve LLM tokens on parse failure`

**Causa-raiz:** 2 turns falharam com `Unterminated string in JSON at
position 2356/2669`. O LLM atingiu o cap de 1000 tokens de output
no meio do JSON, deixando-o truncado. Quando o `JSON.parse` jogou,
o catch externo em [generator/service.ts:282-293](../src/atendente/generator/service.ts) descartava
os tokens já capturados pelo `callOpenAI`, então a telemetria
acabava com `tokens=0/0` e sem chance de confirmar a hipótese.

**Mudança:**
- [`src/atendente/generator/service.ts:216`](../src/atendente/generator/service.ts) — `maxTokens: 1000` → `1500`.
- [`src/atendente/generator/service.ts:207-294`](../src/atendente/generator/service.ts) — `llmResult` agora é declarado com `let` no escopo externo do try, e o catch propaga `input_tokens`, `output_tokens` e `duration_ms` quando disponíveis.
- Tipo `OpenAICallResult` importado de `shared/llm-clients/openai.js`.

**Verificação local:** 26/26 testes do generator passam.

### Fix 2 — Bug F — Proibir metadados de contato como `evidence_text` na Organizadora

**Commit:** [`8f428ad`](https://github.com/financaswall-beep/farejador-pneus/commit/8f428ad)
`fix(organizadora): forbid contact metadata as evidence_text`

**Causa-raiz:** Em [`src/organizadora/prompt.ts:169`](../src/organizadora/prompt.ts) o
próprio prompt builder injeta `"Nome no cadastro: ${contactName}"`
no user content. O LLM copiou essa string inteira como
`evidence_text` em vez de citar a transcrição do cliente. A regra
2 do system prompt só dizia "trecho exato da mensagem que justifica
o fato", e o bloco de metadado **também** estava na user message,
então o LLM achou que era válido.

**Mudança:**
- [`src/organizadora/prompt.ts`](../src/organizadora/prompt.ts) — regra 2 reescrita para exigir que `evidence_text` seja citação **verbatim de uma linha da transcrição** (linhas que começam com `[msg_id: ...]`). Proíbe explicitamente strings `"Nome no cadastro:"` / `"Cidade no cadastro:"` como evidence. Manda dropar o fato inteiro se a única fonte é metadado.

**Verificação local:** 17/17 testes da organizadora passam (3 evidence + 7 prompt + 7 deterministic-facts).

### Fix 3 — Bug G — Proibir `record_offer`/`create_item` em skill não-comercial

**Commit:** [`a366389`](https://github.com/financaswall-beep/farejador-pneus/commit/a366389)
`fix(generator): forbid record_offer/create_item in non-commercial skills`

**Causa-raiz:** Turn 3 da conv `e6c4d94d` teve skill =
`responder_logistica` (cliente perguntou de entrega), mas o
Generator ainda emitiu `create_item` + `record_offer` apontando pra
um produto de turn anterior (`9fdefe94`, R$ 89). O validator
bloqueou com `item_not_found` (Bug A), mas mesmo depois do fix do
A o turn ainda quebraria, porque [`action-validator.ts:120-127`](../src/atendente/validators/action-validator.ts) exige
que cada `record_offer.products[].product_id` esteja nos tool
results do turno corrente — e esse turn só chamou `calcularFrete`.

**Mudança:**
- [`src/atendente/generator/prompt.ts`](../src/atendente/generator/prompt.ts) — regra 14 nova: se `planner_decision.skill ∈ {responder_logistica, responder_geral, escalar_humano}`, está proibido emitir `record_offer` ou `create_item`. Permitido: `update_slot`, `update_draft`, `say`. Também instrui a não repetir dados comerciais de turnos anteriores sem confirmação fresca.

### Fix 4 — Bug C — Stock só com `verificarEstoque`, nunca de `buscarProduto`

**Commit:** [`c67459d`](https://github.com/financaswall-beep/farejador-pneus/commit/c67459d)
`fix(generator): clarify that stock requires verificarEstoque, not buscarProduto`

**Causa-raiz:** Turn 71504f66 (conv `beb61ad4`) foi bloqueado com
`stock_claim_without_verificar_estoque`. O Generator citou "R$ 79 +
10 unidades em estoque". O `buscarProduto.output[0]` realmente
tinha `total_stock_available: 10`, mas
[`tool-results.ts:72-76 hasStockEvidence`](../src/atendente/validators/tool-results.ts) só aceita
`verificarEstoque` como prova de estoque — `total_stock_available`
dentro de `buscarProduto` é tratado como referencial, não garantia.
A regra 2 antiga do prompt era ambígua: "NAO invente estoque. Use
apenas dados de tool_results" não distinguia.

**Mudança:**
- [`src/atendente/generator/prompt.ts`](../src/atendente/generator/prompt.ts) — regra 2 reescrita explicitando que `total_stock_available` de `buscarProduto` NÃO autoriza afirmação de estoque ao cliente. Para citar quantidade/disponibilidade/pronta entrega, exige `verificarEstoque` no turno atual.

### Fix 5 — Bug A — Race `record_offer`+`create_item` no mesmo turno

**Commit:** [`58aca81`](https://github.com/financaswall-beep/farejador-pneus/commit/58aca81)
`fix(validator): accept items created earlier in same turn (incoming_item_ids)`

**Causa-raiz:** 4 turns foram bloqueados com `action_blocked:item_not_found`.
Em todos, o `record_offer.item_id` é idêntico ao `create_item.item_id`
do MESMO array `blocked_actions` (UUIDs: f806aee1, 4d7d5f2a,
d9f6d4f1, f7d8d2d4). O loop em [`generator/service.ts:73-78`](../src/atendente/generator/service.ts)
valida cada action contra `context.state` original, sem
propagar — então `record_offer` no índice 3 do array nunca enxerga
o `create_item` no índice 0.

**Mudança:**
- [`src/atendente/validators/action-validator.ts`](../src/atendente/validators/action-validator.ts):
  - Adicionado `incoming_item_ids?: Set<string>` em `ActionValidationContext`.
  - `itemExists` aceita o set como fallback.
  - `record_offer`, `set_active_item`, `update_item_status` agora aceitam item do set; o check de status (`'aberto'|'ofertado'`) é pulado para item recém-criado (nasce `'aberto'` em [`apply-action.ts:240`](../src/atendente/state/apply-action.ts)).
  - **A segunda checagem de `record_offer` (`offer_product_not_supported_by_tool_result`) continua valendo** — fresh item não ganha free pass no product_id grounding.
- [`src/atendente/generator/service.ts:73`](../src/atendente/generator/service.ts) — antes do loop, coleta `incomingItemIds = new Set(actions.filter(a => a.type === 'create_item').map(a => a.item_id))` e passa no `actionCtx`.
- [`tests/unit/atendente/validators/action-validator.test.ts`](../tests/unit/atendente/validators/action-validator.test.ts) — 5 testes novos cobrindo: in-turn item permitido, id desconhecido ainda bloqueia, set_active_item/update_item_status com item fresco, segunda checagem (product_id) ainda firando.

**Cobertura:** Resolve 3 dos 4 turns de Bug A. O 4º turn (e6c4d94d turn 3, `responder_logistica`) é coberto pelo Fix 3 (Bug G) que impede o Generator de emitir `record_offer` nessa skill.

**Verificação local:** 64/64 testes dos validators passam (10 antigos + 5 novos do action-validator + 49 do say-validator).

### Fix 6 — Bug B — Separar `current_turn_tool_results` de `tool_results_history`

**Commit:** [`c415576`](https://github.com/financaswall-beep/farejador-pneus/commit/c415576)
`fix(generator): split current-turn tool results from history to stop stale prices`

**Causa-raiz:** 6 turns bloqueados com `money_mentioned_without_tool_result`.
Em todos os 6, o preço citado (R$ 79, 89, 99, 120, 149×2) bate
exatamente com o `price_amount` de `buscarProduto` chamado em um
turno **anterior** da mesma conversa. O JSON do user message do
Generator tinha dois campos quase-gêmeos:
- `context.recent_tool_results` (histórico, vem de [`context-builder.ts:140-151`](../src/atendente/planner/context-builder.ts) com `JSON.stringify(payload).slice(0,500)` — preço inclusivo)
- `tool_results` (turno atual, vem de `decision.tool_requests` executados)

O system prompt só dizia "Use apenas valores presentes em tool_results", ambíguo. O say-validator em [`generator/service.ts:50-57 toValidationCtx`](../src/atendente/generator/service.ts) só recebe os tool results do turno atual. Mismatch puro de contrato entre prompt e validator.

**Mudança:**
- [`src/atendente/generator/prompt.ts`](../src/atendente/generator/prompt.ts):
  - JSON do user: `recent_tool_results` → `tool_results_history`; `tool_results` → `current_turn_tool_results`.
  - Regra 1 reescrita: preço/estoque/frete/taxa **só** podem citar valores presentes em `current_turn_tool_results`. `tool_results_history` é memória de turnos passados e NÃO autoriza menção comercial nova. Se `current_turn_tool_results` não tem o valor, NÃO cite valor algum — use fallback seguro.
  - Regras 3 (frete) e 4 (compatibilidade) atualizadas para usar o mesmo nome.
- Validator **não foi mexido** — continua gate único.
